2. Always-on rules (1–7) — hard rules for every task

1 Reuse before build — find and reuse/extend something existing before building new; make the smallest change that fits the existing design
2 No hardcode — settings live on a tools/admin surface + DB, not baked into .env/code; secrets only in gitignored env files behind a prod boot-guard; UI text is i18n (t(key))
3 Registries first — read/update ports.md + versions.md in the same commit
4 Running — ask first — always ask "should I run it, or will you?" → only via a start script / docker compose up -d, never a hidden dev server
5 Docs discipline — change code/structure → update the owning doc + index in the same commit
6 Docs in English — AI-first; local language only for content
7 Code clarity — write for the next reader (KISS); comments explain why; match the file's style

Versioning rules (§6.5)
  • Every copy has an app_version (config.py → /api/health) — not hardcoded
  • UAT bumps freely · main stays at the last promoted version
  • While they differ → the doc carries ## Unreleased — pending promotion
  • Promote only when told → versions converge again
  • Registry = versions.md (UAT / Production / pending)

⚠ Example (measured 2026-06-20): a shared module that diverged across two apps became reconciliation work → extract a shared package early (see §6.4).

When to split a database (database-design.md)

Split when there's a clear seam (bounded context): a stateful standalone (its own DB) · per-department install · independent lifecycle/scaling

Don't split when it would create cross-DB FKs or just "looks cleaner"

If you split: no cross-DB FK · one module owns its tables · separate migration chain · register the port

ER principles: meaningful names · intentional FK + ON DELETE · normalize by default · index what you filter/join on · paginate (keyset)
